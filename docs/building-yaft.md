# Building yaft for reMarkable 2

This document explains how to cross-compile yaft with custom fonts for a reMarkable 2 running firmware **3.3.x** (glibc ~2.30).

## Why this is non-trivial

The reMarkable SDK (`/opt/codex/rm2/5.0.58-dirty`) targets glibc 2.38–2.39, which is incompatible with older firmware. The **toltec toolchain** (`ghcr.io/toltec-dev/toolchain:v3.1`) targets glibc 2.30, which works.

The toltec toolchain is distributed as a Docker image. It doesn't include `libudev` or `libsystemd`, which rMlib requires — those must be injected from the rM SDK sysroot.

---

## One-time setup

### 1. Install prerequisites

```bash
# Docker (for the cross-compiler)
sudo pacman -S docker
sudo systemctl start docker

# The rM SDK must already be installed at /opt/codex/rm2/5.0.58-dirty
# See: https://remarkable.guide/devel/toolchains.html
```

### 2. Pull the toltec Docker image

```bash
docker pull ghcr.io/toltec-dev/toolchain:v3.1
```

### 3. Save the Docker image for offline use

The image is 1.8 GB and takes time to pull. Save it locally:

```bash
docker save ghcr.io/toltec-dev/toolchain:v3.1 | gzip > ~/toltec-toolchain-v3.1.tar.gz
```

To restore it on another machine (or after a `docker system prune`):

```bash
docker load < ~/toltec-toolchain-v3.1.tar.gz
```

### 4. Create the `/tmp/toltec-extra` directory

The toltec toolchain doesn't include `libudev`/`libsystemd`. Copy them from the rM SDK sysroot:

```bash
RM_SYSROOT=/opt/codex/rm2/5.0.58-dirty/sysroots/cortexa7hf-neon-remarkable-linux-gnueabi

mkdir -p /tmp/toltec-extra/usr/{lib/pkgconfig,include/systemd}

# libudev
cp $RM_SYSROOT/usr/lib/libudev.so*   /tmp/toltec-extra/usr/lib/
cp $RM_SYSROOT/usr/include/libudev.h /tmp/toltec-extra/usr/include/

# libsystemd (needed by libudev)
cp $RM_SYSROOT/usr/lib/libsystemd.so* /tmp/toltec-extra/usr/lib/
cp -r $RM_SYSROOT/usr/include/systemd/ /tmp/toltec-extra/usr/include/

# pkg-config files (prefix must match the toltec sysroot path at build time)
TOLTEC_SYSROOT=/opt/x-tools/arm-remarkable-linux-gnueabihf/arm-remarkable-linux-gnueabihf/sysroot/usr

cat > /tmp/toltec-extra/usr/lib/pkgconfig/libudev.pc << EOF
prefix=$TOLTEC_SYSROOT
exec_prefix=$TOLTEC_SYSROOT
libdir=$TOLTEC_SYSROOT/lib
includedir=$TOLTEC_SYSROOT/include

Name: libudev
Description: libudev
Version: 255
Libs: -L\${libdir} -ludev
Cflags: -I\${includedir}
EOF

cat > /tmp/toltec-extra/usr/lib/pkgconfig/libsystemd.pc << EOF
prefix=$TOLTEC_SYSROOT
exec_prefix=$TOLTEC_SYSROOT
libdir=$TOLTEC_SYSROOT/lib
includedir=$TOLTEC_SYSROOT/include

Name: libsystemd
Description: systemd Library
Version: 255
Libs: -L\${libdir} -lsystemd
Cflags: -I\${includedir}
EOF
```

**Note:** `/tmp/toltec-extra` is lost on reboot. Re-run step 4 after each reboot (or move it to a permanent location and update the script variables).

### 5. Build the host binary (needed for the noto font header)

The cross builds borrow the pre-generated noto font header from the host build:

```bash
cmake --preset dev-host
cmake --build build/dev-host --target yaft
```

## Building yaft

### Option A: Build a specific binary (current font)

```bash
docker run --rm \
  -v "$(pwd):/repo" \
  -v /tmp/toltec-extra:/toltec-extra \
  -v /tmp/toltec-cross.ini:/tmp/cross.ini:ro \
  ghcr.io/toltec-dev/toolchain:v3.1 bash << 'EOF'
# ... see scripts/test-fonts.sh for the full inner script
EOF
```

The working full build command is documented in `scripts/test-fonts.sh`.

### Option B: Test multiple font sizes

```bash
./scripts/test-fonts.sh                    # build all variants
./scripts/test-fonts.sh root@10.11.99.1   # build + scp each binary to device
```

This script:
1. Downloads Terminus fonts (from SourceForge) and Spleen fonts (from GitHub) to `/tmp/yaft-fonts/`
2. Uses a BDF 2× pixel scaler (built into the script) to create intermediate sizes
3. Builds libevdev 1.13.1 from source (ARM static) and caches it to `/tmp/toltec-libevdev/`
4. Runs one Docker build per font variant, outputs binaries to `build/font-test/`

**Font variants** (all have Polish support — żółćęśąźń ŻÓŁĆĘŚĄŹŃ):

| Binary | Cell size | Cols × Rows (rM2 1404×1872) |
|--------|-----------|------------------------------|
| `yaft-terminus-16x32` | 16×32 | 87 × 58 (original, small) |
| `yaft-terminus-24x48` | 24×48 | 58 × 39 ★ recommended |
| `yaft-terminus-28x56` | 28×56 | 50 × 33 |
| `yaft-terminus-32x64` | 32×64 | 43 × 29 |
| `yaft-spleen-16x32`   | 16×32 | 87 × 58 |
| `yaft-spleen-32x64`   | 32×64 | 43 × 29 (currently deployed) |

The `terminus-*` variants are generated by 2× pixel-scaling the original Terminus BDF bitmaps.
The `spleen-*` variants use the Spleen font directly.

**Re-running after a reboot** — all `/tmp/` caches are lost. The script regenerates them automatically, but to save time you can persist them:

```bash
# Persist libevdev cache and fonts between reboots
cp -r /tmp/toltec-libevdev ~/toltec-build-cache/libevdev
cp -r /tmp/yaft-fonts       ~/toltec-build-cache/fonts
# Then set these at the top of test-fonts.sh:
FONTS_DIR=~/toltec-build-cache/fonts
LIBEVDEV_CACHE=~/toltec-build-cache/libevdev
```

## Deploying to the device

```bash
# Copy a binary
scp build/font-test/yaft-terminus-24x48 root@10.11.99.1:/home/root/

# Test it (doesn't replace the installed yaft)
ssh root@10.11.99.1 'LD_PRELOAD=/opt/lib/librm2fb_client.so.1.0.1 /home/root/yaft-terminus-24x48'

# Polish character test (run inside yaft)
echo 'żółćęśąźń ŻÓŁĆĘŚĄŹŃ'

# If happy with it, replace the installed binary
scp build/font-test/yaft-terminus-24x48 root@10.11.99.1:/opt/bin/yaft
```

Device connection: USB RNDIS at `10.11.99.1`, SSH alias `remarkable`.

## How the font system works

The font is entirely defined by `vendor/libYaft/glyph.h`. It contains:

```c
struct glyph_t {
    uint32_t code;      // Unicode codepoint
    uint8_t width;      // 1 = half-width, 2 = full-width
    uint64_t bitmap[CELL_HEIGHT];
};
enum { CELL_WIDTH = 32, CELL_HEIGHT = 64 };   // ← controls layout everywhere

static const struct glyph_t glyphs[] = { ... };
static const struct glyph_t bold_glyphs[] = {};  // empty: falls back to regular
```

`CELL_WIDTH` and `CELL_HEIGHT` are used throughout `terminal.cpp` and `screen.cpp` at compile time. Changing the font requires recompiling.

### Generating a new glyph.h from a BDF file

```bash
# Requires upstream yaft's mkfont_bdf tool (built once, stays in /tmp/yaft-upstream)
cd /tmp/yaft-upstream
./mkfont_bdf table/alias /path/to/font.bdf 2>/dev/null | grep -v '^>>WARN' > /tmp/glyph.h
echo 'static const struct glyph_t bold_glyphs[] = {};' >> /tmp/glyph.h
```

The `>>WARN<<` filter is needed because mkfont_bdf writes a warning to stdout (not stderr)
when it can't find a wide-character substitute for U+3000.

### The `screen.cpp` bitmap fix

With `CELL_WIDTH = 32`, the bitmap bit-test needs `uint64_t` arithmetic:

```cpp
// CORRECT (apps/yaft/screen.cpp line 226):
(glyph->bitmap[h] & (static_cast<uint64_t>(1) << (bdfPadding + CELL_WIDTH - 1 - w))) != 0U

// WRONG — UB when shift reaches 31 with signed int:
(glyph->bitmap[h] & (0x01 << (bdfPadding + CELL_WIDTH - 1 - w))) != 0U
```

This fix is already applied in the repo.

## Troubleshooting

**`GLIBC_2.32 not found` on device**
→ You used the rM SDK toolchain (`cmake --preset dev`). Use the toltec Docker build instead.

**`cannot find -levdev`**
→ libevdev must be built as a static library (`--default-library=static` in meson). The `test-fonts.sh` script handles this automatically.

**`set_target_properties called with incorrect number of arguments`**
→ Known cmake bug when `LIBEVDEV_LIBRARIES` has multiple entries (e.g., `evdev;rt`). Fixed in `vendor/evdev/CMakeLists.txt` by quoting `"${LIBEVDEV_LIBRARIES}"`.

**`XXD not found, required for font`**
→ The toltec container lacks `xxd`. The build scripts install a Python replacement before calling cmake.

**`>>WARN<< couldn't find wide substitute` in glyph.h**
→ mkfont_bdf printed a warning to stdout. Filter it: `./mkfont_bdf ... | grep -v '^>>WARN'`.

**Scaled font shows wrong CELL_HEIGHT**
→ mkfont_bdf computes height from `FONT_ASCENT + FONT_DESCENT`, not `FONTBOUNDINGBOX`. The 2× scaler in `test-fonts.sh` doubles all four: `SIZE`, `FONT_ASCENT`, `FONT_DESCENT`, and `FONTBOUNDINGBOX`.
